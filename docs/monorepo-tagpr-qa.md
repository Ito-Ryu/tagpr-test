# tagpr モノレポ運用におけるマージ手法の Q&A （AIとの会話）

## Q. なぜモノレポでは「Squash and merge」が必須なのですか？単一リポジトリの時は「Create a merge commit」でも問題なく動いていたのはなぜですか？

**A.**
結論から言うと、`tagpr` が内部で実行している **Git コマンドの「検索条件（引数）の違い」** と、Git に備わっている **「History Simplification（履歴の単純化）」** という仕様が組み合わさることで、マージコミットが検索結果から消滅してしまうためです。

詳細な理由は以下の通りです。

### 1. モノレポ「ではない」（単一リポジトリ）時の挙動
単一リポジトリ（`.tagpr` に `tagPrefix` を設定していない状態）の場合、`tagpr` はリポジトリ全体の Pull Request を探すため、以下のようなシンプルな Git コマンドを裏側で実行します。

```bash
git log --merges origin/main
```

この時、Git は **「リポジトリ全体のすべてのマージコミット」** を素直に出力してくれます。そのため、普通に「Create a merge commit」でマージしても、`tagpr` は確実にそのマージコミットを見つけ出し、紐づく PR のラベル（`major` など）を正しく読み取ることができます。

### 2. モノレポの時の挙動（tagPrefix を設定した場合）
一方、モノレポ構成（例として `tagPrefix = app1` を設定している状態）の場合、`tagpr` は「他のアプリの変更を誤って拾わないようにする」ため、検索コマンドの末尾に **`-- app1`（パス指定）** を付け足して実行します。

```bash
git log --merges origin/main -- app1
```

ここで、Git に組み込まれている **「History Simplification（履歴の単純化）」** という強力な最適化機能が発動します。

Git は、パス指定（`-- app1`）付きでログを検索された際、次のように判断します。
> 「このマージコミットは、コンフリクトも起きていないし、ただ `app1` フォルダの変更をメインブランチにそのまま取り込んだだけの綺麗なマージだね。なら、わざわざ『マージしたよ』というコミット自体は検索結果に表示しなくても、中身のコミットだけ見せれば履歴としては十分だよね」

その結果、Git はマージコミットを検索結果から意図的に**消し去って（非表示にして）**しまいます。
マージコミットが検索結果に出てこなくなるため、`tagpr` は「app1 に関するマージPRは1つもない」と誤認してしまいます。当然、PRに付与されていた `major` などのラベルも読み取れないため、デフォルトの Patch アップデートとして処理してしまいます。

### 結論と対策
*   **単一リポジトリ**: パス指定がないため、Git が最適化を行わず、すべてのマージコミットが見える。だから通常のマージでも動く。
*   **モノレポ**: パス指定（`-- app1`など）があるため、Git が気を利かせて「綺麗なマージコミット」を検索結果から隠してしまう。だから `tagpr` が PR を見失う。

これが、モノレポ構成の時にだけ「Squash and merge」が必須になる技術的な理由です。

**対策:**
「Squash and merge」を使用すると、マージコミットではなく「内容がまとまった1つの通常のコミット」として main ブランチに乗るため、Git の最適化によって隠されることなくパス指定検索でも確実にヒットします。
これにより `tagpr` が PR を正しく発見し、ラベルを読み取って意図通りのバージョンアップを行えるようになります。誤操作を防ぐため、リポジトリの設定で **「Allow squash merging」のみを有効にしておく** ことを強く推奨します。

## Q. 通常マージ（Create a merge commit）の場合、バージョンは Patch しか上がらないのに、なぜ CHANGELOG のテキストの中にはちゃんと PR のタイトルが載っているのですか？

**A.**
これは、`tagpr` が内部で **「バージョンを決定する仕組み」と「CHANGELOGの文章を生成する仕組み」の2つに全く異なる技術を使っている** ことによって生じる「ねじれ現象」です。

### 1. バージョンの決定（ローカルの Git コマンド）
`major` や `minor` などのラベルを検知してバージョンを決定する処理は、GitHub Actions のランナー上で動く **ローカルの `git` コマンド** によって行われます。
先述の通り、`git log --merges -- app1` のようにパス指定で検索すると、Git の `History Simplification`（履歴の単純化）仕様によりマージコミットが隠されてしまいます。
その結果、ローカルでの検索では PR を見つけられず、当然 `major` ラベルも読み取れないため、`tagpr` はデフォルトの **Patch アップデート** を決定してしまいます。

### 2. CHANGELOG の生成（GitHub の API）
一方で、CHANGELOG の文章（Markdown）を生成する際、`tagpr` はローカルの Git コマンドを使わず、**GitHub 公式の「リリースノート自動生成 API (`GenerateReleaseNotes`)」** に処理を依頼します。

GitHub のサーバー側（API）は、ローカルの `git` コマンドが持つ「履歴を単純化して隠す」というお節介な仕様を持っていません。
すべてのコミットと PR の関係を正確に把握しているため、通常マージされた PR であっても確実に見つけ出します。そして、その PR に付いている `app1` などのラベルを見て `release.yaml` のルールに合致すると判断し、**CHANGELOG の文章内に正しく項目を生成**して返してくれます。

### 結論
*   **バージョンの決定**: ローカルの `git` コマンドに依存しているため、Git の仕様によりマージコミットを見失い、**失敗して Patch になった。**
*   **CHANGELOGのテキスト**: 優秀な GitHub API に依存しているため、通常マージの PR も確実に見つけ出し、**成功して文章に載った。**

このように、裏側で使われている技術が異なっていたために「文章には載っているのにバージョンは上がっていない」という矛盾した結果になっていました。
このねじれを解消し、ローカルの Git 検索も確実に成功させるための唯一の手段が「Squash and merge」の利用となります。
